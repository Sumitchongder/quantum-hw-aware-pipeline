# Contributing to Quantum Hardware-Aware Co-Design Pipeline

Thank you for your interest in contributing to the **Hardware-aware Low-latency Quantum Compilation and Data-driven QED Scheduling Framework**. This project provides a reproducible artifact layer for cross-layer quantum systems research targeting Early Fault-Tolerant Quantum Computing (EFTQC) regimes.

To maintain the rigorous scientific and engineering standards required for high-impact Q1 journal venues and peer-reviewed artifact evaluations (ACM/IEEE/Springer), all contributions must comply with the guidelines outlined below.

---

## 1. Code of Conduct & Academic Integrity
By contributing to this repository, you agree to uphold strict scientific transparency and academic integrity. Fabricating benchmark metrics, suppressing trade-offs (such as post-selection state retention penalties), or masking classical compilation latency overheads is strictly prohibited.

---

## 2. Research Development Workflow

We follow a structured branch-and-merge architecture. Please follow these steps to propose enhancements:

1. **Fork the Repository**: Create a personal copy of the repository on GitHub.
2. **Create a Feature Branch**: Isolate your changes inside a descriptive branch.
   ```bash
   git checkout -b feature/expanded-noise-models

```

3. **Set Up the Hermetic Environment**: Ensure you are using the precise containerized or HPC-module toolchain specified in the `docker/` configuration:
```bash
./scripts/run_docker.sh

```


4. **Implement Extensions**: Ensure changes align mathematically with our decoupled optimization abstractions (Global Routing, Local ILP Refinement, or ML-Driven Scheduling).
5. **Enforce Local CI Sanity**: Before submitting a Pull Request, run the local verification workflow to guarantee that your modifications preserve circuit topology and pass performance constraints:
```bash
# Run formatting tools
black .
isort --profile black .

# Run static linting analysis
flake8 . --count --select=E9,F63,F7,F82 --show-source --statistics

# Run regression testing matrix
pytest tests/ -v

```


6. **Submit a Pull Request (PR)**: Target your PR against the `main` or `master` branch. Include a comprehensive description detailing the algorithmic changes, mathematical formulation, and the empirical impact on fidelity or execution latency.

---

## 3. Core Technical Standards

### A. Algorithmic Changes & Mathematical Rigor

If your contribution modifies the compiler pass or the scheduling policy:

* You **must** provide a corresponding update to the asymptotic complexity documentation.
* New compilation passes cannot introduce overheads that violate real-time control constraints ($< 6\text{ ms}$ inference bounds).
* Any new scheduling model (e.g., Graph Neural Networks, Reinforcement Learning) must match or exceed the microsecond-level latency performance of our baseline XGBoost scheduler.

### B. Unit & Integration Testing

Every new feature requires explicit testing scripts appended to the `tests/` directory:

* **Routing passes** must verify layout mapping invariance (the output circuit must structurally conform to the target hardware coupling map).
* **Syndrome insertion layers** must include assertions validating that data qubit states are structurally preserved and that ancilla lines are recycled safely using immediate `reset` operations.

### C. Benchmarking and Metrics Integrity

When adding a new quantum benchmark circuit or noise topology profile:

* Evaluate scales across the full tracking horizon (**6 to 40 qubits**).
* Report metrics utilizing non-parametric statistical significance evaluations (such as the Wilcoxon signed-rank test or Mann-Whitney U test) reporting explicit $p$-values ($p < 0.01$).
* Include standard error margins or **95% Confidence Intervals** on all comparative fidelity charts.
* Document physical reality-gap variables if validating against live physical hardware (QPU specifications, calibration dates, total shot metrics).

---
